# XFA Dynamic Forms — Implementation Report

**Date:** 2026-06-19
**Scope:** `forms::reader` (XFA detection/extraction), WASM bridge, REST API

## What Was Implemented

- `forms::reader::is_xfa_form(doc: &PdfDocument) -> Result<bool>` — detects an
  XFA form via `/AcroForm /XFA`. Ungated (used as a pre-check before other
  calls, including by Free-tier callers who need to know to route elsewhere).
- `forms::reader::extract_xfa_data(doc: &PdfDocument) -> Result<String>` —
  extracts the raw XFA XML. Handles both the single-stream and
  alternating-`[name, stream]`-packet-array forms of `/XFA` (ISO 32000-2
  §12.7.8). Gated `Tier::Enterprise` via `license::require` (behind
  `#[cfg(feature = "crypto")]`, matching the existing convention in
  `forms::fdf`/`forms::filler`).
- `forms::reader::get_acroform_dict` (private) — shared resolution helper for
  both functions, using `Catalog::acroform()` rather than re-deriving the
  `/Root /AcroForm` walk already done in `read_form_fields`.
- Re-exported `is_xfa_form`, `extract_xfa_data` from `forms::mod`.
- WASM (`wasm/editor.rs`): `WasmEditor::is_xfa_form`, `extract_xfa_data`,
  and `flatten_xfa_or_error` (detects XFA and returns a guidance `JsError`
  pointing at the REST endpoint instead of silently no-op flattening an
  AcroForm-only document).
- REST API (`server/handlers.rs` + `server/routes.rs`):
  - `POST /api/v1/form/xfa/detect` → `{"is_xfa": bool}`
  - `POST /api/v1/form/xfa/extract` → raw XFA XML (`text/xml`), 422 if absent
  - `POST /api/v1/form/xfa/flatten` → flattened PDF bytes, via
    `flatten_xfa_via_libreoffice` (shells out to `libreoffice`, falling back
    to `soffice`, headless `--convert-to pdf`)

## Design Decisions

- **Tier gating only on `extract_xfa_data`, not `is_xfa_form`.** Detection
  needs to be available to all callers (including Free tier) so they can
  decide to route to the server or surface a clear error; only the actual
  data extraction is an Enterprise capability per the phase doc.
- **No `tempfile` dependency added.** Built a unique temp path from
  `std::env::temp_dir()` + PID + a nanosecond timestamp instead of pulling in
  a new crate for one call site. Input and output paths are intentionally
  the same (`<dir>/pdf-core-xfa-<pid>-<nonce>.pdf`) — LibreOffice's
  `--convert-to pdf --outdir <dir>` on a `.pdf` input overwrites that same
  path with the converted (flattened) output, which is fine since the read
  happens strictly after the process exits.
- **`libreoffice` with `soffice` fallback.** Some distros only ship the
  `soffice` binary name; the handler tries `libreoffice` first, then retries
  with `soffice` if spawning the first one fails (binary not found), rather
  than hard-failing on naming differences.
- **WASM `flatten_xfa_or_error` does not call `self.save()`.** Matches the
  existing convention of `flatten_form_fields`/`flatten_all_form_fields`,
  which mutate `self.editor` in place and let the JS caller fetch bytes
  separately — the phase doc's draft (`self.save()` returning `Vec<u8>`) did
  not match the actual `WasmEditor` API and was not followed literally.
- **No `pdf_inspect`/CLI exposure.** Out of scope per the phase doc, which
  only specifies library, WASM, and REST surfaces.

## Test Coverage

All in `forms::reader::tests` (12 new tests):

- `is_xfa_form_false_when_no_acroform` — no `/AcroForm` at all
- `is_xfa_form_false_when_acroform_has_no_xfa` — `/AcroForm` present, no `/XFA`
- `is_xfa_form_true_when_xfa_key_present` — `/XFA` as a single stream
- `extract_xfa_data_errors_when_no_acroform` — error path
- `extract_xfa_data_errors_when_no_xfa_key` — error path
- `extract_xfa_data_single_stream` — happy path, single-stream `/XFA`
- `extract_xfa_data_packet_array_concatenates_in_order` — happy path,
  `[name, stream, name, stream]` packet array, asserts concatenation order

Verified: `cargo fmt --check`, `cargo clippy --features server,wasm -- -D
warnings` (clean), `cargo test --features forms,crypto,server,wasm --lib
--bins` (646 passed), `cargo build --target wasm32-unknown-unknown
--no-default-features --features forms,crypto --lib` (the XFA-touched code
builds clean for wasm32).

## Known Limitations / Follow-up

- **Full `wasm` feature build on `wasm32-unknown-unknown` could not be
  verified end-to-end in this environment.** `rquickjs-sys`'s vendored
  QuickJS source patch (`patches/error_column_number.patch`) fails to apply
  in this checkout — confirmed pre-existing and unrelated to this change via
  `git stash` (identical failure on unmodified `main`). The XFA-specific
  code was independently verified to build for wasm32 via
  `--no-default-features --features forms,crypto`.
- **`cargo test` for the full `wasm` feature set fails on
  `tests/wasm_api.rs`** (`unresolved imports ... no WasmDocument/WasmEditor/
  WasmPdfWriter in wasm`) — also pre-existing (reproduced on unmodified
  `main`); that test file is gated for the `wasm32` target and doesn't build
  natively regardless of this change.
- **No CI/integration test for the LibreOffice round-trip.** Doing so would
  require a `libreoffice`/`soffice` binary in the test environment and a
  real XFA fixture PDF; out of scope without one to hand. The handler logic
  was reviewed but not exercised against a real binary in this session.
- **No mobile FFI bridge for XFA.** The phase doc only specifies
  reader/WASM/REST surfaces; `ffi`/`mobile` exposure was not requested and
  was not added.
