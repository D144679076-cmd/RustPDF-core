# crypto-aes256 — Implementation Report

**Date:** 2026-06-05
**Scope:** AES-256 (R5/R6) PDF decryption — Phase 1

## What Was Implemented

- **`src/crypto/aes256.rs`** — fixed integer overflow in `compute_hash_r6`: the termination check `last_byte + 32 <= round as u8` overflowed for `last_byte > 223`; corrected to `last_byte as u32 + 32 <= round`.
- **`tests/gen_aes256_fixture.py`** (new file) — standalone Python script using the `cryptography` library that generates `tests/fixtures/encrypted_aes256.pdf` (AES-256 R6, user password `"test"`). Implements Algorithm 2.B, `/U`/`/UE`/`/O`/`/OE`/`/Perms` construction per ISO 32000-2 §7.6.4.
- **`tests/fixtures/encrypted_aes256.pdf`** (new file) — fixture produced by the above script.
- **`tests/real_pdf.rs`** — added `aes256_encrypted_pdf_opens_with_password` and `aes256_encrypted_pdf_wrong_password_fails` integration tests (both `#[cfg(feature = "crypto")]`); updated `all_fixtures_parse_successfully` to skip `PdfError::Encrypted` errors rather than panic.

## Design Decisions

- **Inlined `/Encrypt` in trailer**: The `EncryptionHandler::from_trailer` function receives the raw trailer dict and has no access to the XRef table for indirect-reference resolution at that stage. Inlining the encrypt dictionary directly in the trailer (valid per ISO 32000-2 §7.5.5) avoids a larger refactor while keeping the test fixtures correct.
- **Algorithm 2.B overflow fix**: The spec compares `E[last] + 32 ≤ round` where all values are logically integers of unrestricted range; Rust's `u8` wraps silently in release builds but panics in debug. Widening to `u32` is the minimal correct fix.
- **No changes to `handler.rs` decrypt paths**: The `decrypt_string` / `decrypt_stream` AES-256 arms were already present. The phase plan's removal of the early `Err(Encrypted)` return was also already done. This report covers the remaining gap: fixture + tests + overflow fix.

## Test Coverage

| Test | What it covers |
|------|---------------|
| `aes256_encrypted_pdf_opens_with_password` | Happy path: R6 password verification, file-key derivation, document parse, page count |
| `aes256_encrypted_pdf_wrong_password_fails` | Error path: wrong password returns `Err` |
| `all_fixtures_parse_successfully` (updated) | Encrypted fixtures are skipped rather than causing a false failure |

## Known Limitations / Follow-up

- **Indirect `/Encrypt` references**: Real PDFs produced by tools like qpdf use `/Encrypt 4 0 R` (indirect reference) in the trailer. The current `from_trailer` only handles inline dictionaries. A follow-up task should resolve the reference using the XRef before calling `from_trailer`.
- **`/Perms` not validated**: The Rust handler doesn't verify the `/Perms` field. This is optional for correctness but required for full ISO 32000-2 compliance.
- **No encrypted string/stream round-trip test**: The fixture has no string or stream objects that are actually encrypted (only PDF names in the page tree). A future fixture with text content would exercise `decrypt_string` / `decrypt_stream` end-to-end.
