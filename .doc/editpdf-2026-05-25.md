# EditPdf — Implementation Report

**Date:** 2026-05-25
**Scope:** Full edit-PDF function expansion (page ops, content drawing, WASM bindings)

## What Was Implemented

### Page Manipulation (`src/editor/page_editor.rs`)

- `move_page(editor, from_index, to_index)` — reorder pages within the document by removing from one position and inserting at another.
- `rotate_page(editor, index, degrees)` — set page rotation (multiples of 90). Uses `rem_euclid` for normalization.
- `set_crop_box(editor, index, rect)` — set or replace the `/CropBox` on a page to restrict the visible region.

### Content Drawing (`src/editor/content_draw.rs` — NEW)

High-level drawing helpers that open a `ContentLayer`, emit operators, and commit:

- `draw_text(editor, page_index, x, y, text, style)` — single-line text with standard font registration in `/Resources/Font`.
- `draw_rect(editor, page_index, x, y, w, h, style)` — filled, stroked, or both.
- `draw_line(editor, page_index, x1, y1, x2, y2, color, line_width)` — straight line.
- `draw_ellipse(editor, page_index, cx, cy, rx, ry, style)` — Bézier-approximated ellipse (kappa=0.5522847498).
- `place_image(editor, page_index, x, y, dw, dh, pixels, w, h, channels)` — raw pixel image (Gray/RGB/CMYK) via FlateDecode.
- `place_jpeg(editor, page_index, x, y, dw, dh, jpeg_data, pw, ph)` — JPEG pass-through via DCTDecode.

Supporting helpers:
- `register_font_resource(editor, page_id, key, font_id)` — ensures `/Resources/Font` dict has the entry.
- `register_xobject_resource(editor, page_id, key, obj_id)` — ensures `/Resources/XObject` dict has the entry.

Types:
- `TextStyle { font_name, font_size, color }` — parameters for `draw_text`.
- `RectStyle { fill, stroke, line_width }` — parameters for `draw_rect` and `draw_ellipse`.

### WASM Module Split (R4 compliance)

`src/wasm/mod.rs` (was 838 lines) split into:
- `src/wasm/mod.rs` (~80 lines) — logging init + shared helpers.
- `src/wasm/document.rs` (~230 lines) — `WasmDocument`, `WasmRenderer`, `RenderResult`.
- `src/wasm/editor.rs` (~480 lines) — `WasmEditor`, `WasmPdfWriter` + all new bindings.

## Design Decisions

- **Separate `content_draw.rs` from `page_editor.rs`**: Page structure operations (add/delete/move/rotate) are distinct from content-stream drawing. Keeping them in separate files follows R4 (one concern per file) and keeps both under 600 lines.
- **Font registration per draw call**: Each `draw_text` call writes a fresh font object and registers it. This is simple and correct for incremental updates. Deduplication (checking if the font already exists) is deferred — the PDF spec allows multiple font objects with the same BaseFont.
- **Ellipse via 4 cubic Béziers**: The standard kappa approximation (0.5522847498) gives <0.03% error. No dependency needed.
- **`RectStyle` reused for ellipse**: Same fill/stroke/line_width semantics apply to any closed shape.
- **Negative color = no fill/stroke in WASM**: The WASM `draw_ellipse` uses negative R value as a sentinel for "no fill" or "no stroke" since `wasm_bindgen` doesn't support `Option` in flat parameter lists.
- **WASM split preserves public API**: All existing JS-facing types (`WasmDocument`, `WasmEditor`, `WasmPdfWriter`, `WasmRenderer`) keep the same names and methods. The split is internal only.

## Known Limitations / Follow-up

- **Font deduplication**: Multiple `draw_text` calls with the same font create duplicate font objects. A font cache keyed by name would reduce file size.
- **Multi-line text**: `draw_text` handles a single line only. Multi-line layout (word wrap, leading) is not implemented.
- **Image alpha channel**: `place_image` does not support RGBA (4-channel with alpha). Only opaque Gray/RGB/CMYK.
- **Pre-existing `name_tree` test failures**: 4 tests in `document::name_tree` fail due to incorrect xref offsets in hand-crafted PDF bytes. Unrelated to this work.
- **TrueType font embedding via `draw_text`**: Only the 14 standard PDF fonts are supported. Embedding custom TTF fonts requires the existing `write_truetype_font` path, which is not wired into `draw_text` yet.
