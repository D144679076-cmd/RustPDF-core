# display — Implementation Report

**Date:** 2026-05-25
**Scope:** `display` module (`src/display/mod.rs`)

## What Was Implemented

- `DisplayList` — flat, ordered list of drawing commands for one PDF page; built via `DisplayList::from_page(&doc, &page)`
- `DisplayItem` — enum with four variants: `StrokePath`, `FillPath`, `DrawText`, `DrawImage`
- `StrokeStyle` — snapshot of stroke-relevant graphics state (color, line width, cap, join, miter, dash, alpha)
- `FillStyle` — snapshot of fill-relevant graphics state (color, alpha)
- `TextItem` — renderer-ready text span (text, x, y, font_size, font_name, width, color, ctm, alpha)
- `ImageItem` — decoded image with CTM, blend mode, and alpha
- `DisplayListBuilder` — private `OutputDevice` impl that accumulates items during content stream interpretation
- Convenience iterators: `DisplayList::text_items()`, `DisplayList::image_items()`

## Design Decisions

- **Flat list over tree**: a flat `Vec<DisplayItem>` in paint order is the simplest structure a renderer can consume — no tree traversal needed.
- **State snapshots, not references**: each item owns a copy of the relevant graphics state at the time of the draw call. This makes the list self-contained and decouples it from the interpreter lifetime.
- **`DisplayListBuilder` is private**: callers only interact with `DisplayList::from_page`; the builder is an implementation detail.
- **Empty path/text filtering**: the builder skips empty paths and empty text spans to keep the list clean.
- **`blend_mode` stored as `String`**: avoids re-exporting the `BlendMode` enum at the display layer; renderers can match on the string directly.

- `test_stroke_style_from_state` — StrokeStyle fields copied correctly from GraphicsState

- `test_text_item_from_span` — TextItem fields match source TextSpan

## Known Limitations / Follow-up

- `ImageItem.data` is raw decoded bytes with no width/height metadata; the image XObject dict (containing `/Width`, `/Height`, `/ColorSpace`) is not yet captured. A follow-up could add `width: u32`, `height: u32`, `color_space: String` fields.
- Clipping paths are tracked in `GraphicsState.clip_path` but not surfaced in `DisplayItem` — a `ClipPath` variant could be added when a renderer needs it.
- `blend_mode` is stored as a `Debug`-formatted string; a dedicated `BlendMode` re-export would be cleaner for renderer consumers.
