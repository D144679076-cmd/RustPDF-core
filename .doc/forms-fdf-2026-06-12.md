# forms-fdf — Implementation Report

**Date:** 2026-06-12
**Scope:** FDF / XFDF Import & Export (Phase 2)

## What Was Implemented

### New file: `src/forms/fdf.rs`

Public API:
- `export_fdf(doc: &PdfDocument) -> Result<Vec<u8>>` — serialises all AcroForm field values as a valid FDF 1.2 file (complete with `xref` table and `startxref` so the output is parseable by the standard PDF parser).
- `import_fdf(editor: &mut PdfEditor, fdf_bytes: &[u8]) -> Result<()>` — parses FDF bytes via the PDF parser, then dispatches each field to `set_text_field`, `set_checkbox`, or `set_combo_or_list`.
- `export_xfdf(doc: &PdfDocument) -> Result<String>` — serialises all field values as an XFDF 1.0 XML document.
- `import_xfdf(editor: &mut PdfEditor, xfdf_str: &str) -> Result<()>` — hand-rolled state-machine XFDF parser, same dispatch as `import_fdf`.

Private helpers:
- `fdf_value_for_field` — `/Yes`/`/Off` for checkboxes; `(escaped_string)` for others.
- `escape_pdf_string` / `xml_escape` / `xml_attr_escape` / `xml_unescape` — character escaping.
- `parse_fdf` — uses `PdfDocument::parse` to parse FDF, then walks `/Root → /FDF → /Fields`.
- `parse_xfdf` — state machine over the XML string; extracts `name` attribute and `<value>` content for each `<field>` element.

### Modified: `src/forms/mod.rs`
Added `pub mod fdf` and `pub use fdf::{export_fdf, export_xfdf, import_fdf, import_xfdf}`.

### Modified: `src/wasm/editor.rs`
Added four `#[wasm_bindgen]` methods on `WasmEditor`: `export_fdf`, `import_fdf`, `export_xfdf`, `import_xfdf`.

## Design Decisions

**Valid FDF with xref table**: The phase spec's `export_fdf` sketch omitted the `xref` table and `startxref`, which would make the output unparseable by the reused PDF parser. We track byte offsets during serialisation and emit a proper two-entry xref table, making the output valid both as an interchange file and as input to `parse_fdf`.

**PDF parser reuse for FDF import**: FDF is a strict PDF subset. The existing `PdfDocument::parse` handles it without modification — the header check is not enforced (the parser only looks for `startxref`). This avoids a separate FDF tokeniser.

**Pattern matching instead of `into_dict()`/`into_array()`**: The phase spec called `into_dict()` and `into_array()` methods that do not exist on `PdfObject`. The code uses explicit `match` arms consistent with the rest of the codebase.

**Hand-rolled XFDF parser**: No external XML dependency. Advances `pos` past each `</field>` closing tag to handle multiple fields correctly without re-matching the same element.

**License gate on exports only**: `export_fdf` and `export_xfdf` call `crate::license::require(Tier::Pro, ...)` directly. Import functions rely on the per-field gates inside `set_text_field`, `set_checkbox`, and `set_combo_or_list` (consistent with how `set_field_value` in `wasm/editor.rs` works).

## Known Limitations / Follow-up

- **Hierarchical XFDF fields**: The XFDF parser handles flat field lists only. Nested `<field>` elements (representing field groups in the PDF hierarchy) are not supported; each child field must be identified by its `full_name` (dot-joined path).
- **Non-Latin field values in FDF**: `export_fdf` encodes values as raw UTF-8 bytes inside a PDF literal string. PDF viewers expecting PDFDocEncoding for non-ASCII may misinterpret values. A future improvement could emit UTF-16BE strings for non-ASCII content.
- **Multi-select list fields**: FDF/XFDF can express multi-value list selections via an array `/V`, but the current `set_combo_or_list` only accepts a single string. Multi-select support is deferred.
- **`/Kids` in FDF**: Some FDF generators use `/Kids` to represent hierarchical field groups inside the `/Fields` array. The current `parse_fdf` only handles flat arrays of leaf dicts.
